# 案例：Tesco 超市自动购物

**一句话总结**：一个 AI Agent 完全通过浏览器操控（不调用任何 API）自动完成了从餐食规划到超市配送预约的全流程，展示了"无 API 自动化"的巨大潜力。

**人物**：@marchattonhere
**案例**：通过 OpenClaw 浏览器控制实现全自动超市购物

## 完整流程

Agent 接管了一周购物的完整链条：

1. **餐食规划**：根据用户偏好和饮食习惯，自动规划一周七天的三餐菜单
2. **食材清单生成**：根据菜单自动计算所需食材种类和数量，考虑已有存货
3. **Tesco 网站操作**：打开 Tesco 在线超市，逐一搜索和添加食材到购物车
4. **价格优化**：在可选替代品中选择性价比最优的选项
5. **配送预约**：选择合适的配送时间段并完成下单

**关键约束：不使用任何 API，纯浏览器操控。** Agent 像一个真人用户一样点击、滚动、输入、选择——所有操作都发生在 Tesco 的标准网页界面上。Tesco 案例的核心是 [[Chrome DevTools Protocol]] 浏览器控制——AI 像人一样操作网页，这项底层协议让 Agent 突破了"没有 API 就无法自动化"的传统限制。

## 技术要点

这个案例的特殊之处在于：完全依赖 Tool Use 机制 中的浏览器控制能力，而非 API 集成。这意味着理论上可以自动化任何有网页界面的服务，无需等待官方 API 支持。

### 为什么"无 API"如此重要？

传统的自动化方式（如 Zapier、IFTTT）依赖于服务提供商开放 API。但现实中：
- **绝大多数服务没有公开 API**——Tesco 的在线超市就是一个典型例子
- 即使有 API，往往**功能受限**（不支持完整的购物流程）
- API 接入通常需要**开发者账号、审批流程、费用**

浏览器控制彻底绕过了这些障碍。[[AI Agent 定义与分类|Agent]] 看到什么用户就能看到什么，能做用户能做的一切操作。这让自动化的适用范围从"有 API 的服务"扩展到"有网页的一切服务"。这也体现了 Agentic AI 与传统自动化的本质区别——Agent 执行循环赋予了系统适应性和决策能力。

### 技术挑战

- **动态页面处理**：Tesco 网站使用大量 JavaScript 动态渲染，Agent 需要正确等待页面加载
- **反机器人检测**：在线超市通常有防爬虫机制，浏览器控制需要模拟真实用户行为
- **视觉理解**：Agent 需要识别商品图片、价格标签、库存状态等视觉元素
- **状态管理**：从购物车添加到结账是一个多步骤有状态流程，任何一步失败都需要恢复

## 核心洞察

这个案例代表了 AI Agent 自动化的一个重要范式转变——从 **"API 优先"到"界面优先"**。

传统软件集成的逻辑是：先有 API，才能自动化。浏览器控制反转了这个逻辑：只要有网页界面，就能自动化。这对普通用户意义重大——他们不需要理解 API、OAuth、Webhook 这些概念，只需要告诉 Agent "帮我在 Tesco 买这周的菜"。这是 [[编程民主化]] 和 [[可及性突破]] 在日常生活领域的体现，也展示了 自主执行与人机协作 如何让非技术用户受益。

值得注意的风险（详见 [[安全边界与风险（总览）]]）：
- **脆弱性**：网站改版可能导致自动化流程失效
- **速度**：浏览器操控比 API 调用慢一个数量级
- **可靠性**：网络延迟、页面渲染异常都可能导致失败

但对于日常生活自动化场景，这些代价是可以接受的——毕竟没有人需要每秒下 100 单超市订单。

## 相关案例对比

与 [[案例-汽车购买谈判省4200美元]] 同属浏览器自动化类案例，但两者侧重不同：
- Tesco 案例侧重**重复性流程自动化**（每周都要买菜）
- 汽车谈判案例侧重**策略性交互**（需要谈判技巧）

与 [[案例-华尔街级股票筛选器]] 形成互补：后者走的是 API 集成路线（Yahoo Finance + SEC EDGAR），而 Tesco 案例证明即使没有 API 也能完成复杂自动化。两种路线各有适用场景，共同构成了 OpenClaw 自动化能力全景。与 Google Project Mariner 的浏览器自动化相比，OpenClaw 的优势在于能完成购买操作（Mariner 明确不能完成购买、接受 Cookie 等），但缺乏 Mariner 83.5% WebVoyager 基准的标准化测试验证。
