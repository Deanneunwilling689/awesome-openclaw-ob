# 自我修改软件机制

## 一句话总结

OpenClaw 的 Agent 知道自己的源代码在哪里、结构是什么、如何修改——它可以在运行时编辑自己的 Skill 文件和系统提示，修改后 250ms 内系统自动热重载新版本，无需重启。

## 核心机制：Agent 对源代码的感知与修改

Steinberger 的设计哲学：

> "I made the agent very aware -- it knows what its source code is."

这不是一般的"AI 写代码"——Agent 被赋予了对**自身代码**的感知能力。它知道：
- 自己的源文件在哪里（工作空间路径）
- 代码结构是什么（Skill 定义、系统提示模板、工具实现）
- 如何修改才能改变自身行为

### 可修改的范围

Agent 可以直接修改工作空间中的 `.ts` / `.js` 源文件：

| 修改对象 | 效果 |
|----------|------|
| Skill 定义 | 改变 Agent 的能力和行为模式 |
| [[System Prompt 设计\|系统提示]]模板 | 调整 Agent 的人格、规则和指令 |
| 工具实现 | 添加新工具或修改现有工具的逻辑 |

## Skill 热重载：250ms Debounce

文件变更后 **250ms debounce** 自动触发热重载：

```
Agent 修改 Skill 文件
  → 文件系统 watcher 检测到变更
    → 250ms debounce 等待（避免连续保存触发多次重载）
      → 自动加载新版本 Skill
        → 下次 Agent 循环使用新版本
```

**250ms debounce** 的设计意图：Agent 可能在短时间内连续修改多个文件，debounce 确保只在修改完成后触发一次重载，避免加载中间状态。

## 与 Plugin 系统的关系及安全风险

自我修改和[[Plugin 扩展系统]]是两条不同的扩展路径：自我修改由 Agent 自身触发，修改范围限于工作空间内的 Skill 和提示，通过文件 watcher + 250ms debounce 加载，风险最高；Plugin 由开发者手动安装，通过 loader.ts 启动时加载，风险较可控。

**安全隐患**：自我修改能力与[[Prompt Injection 风险]]结合产生严重风险——攻击者可通过提示注入诱骗 Agent 修改 SOUL.md（身份文件），修改后重启仍生效形成**持久化后门**。CEO Matvey Kukuy 演示过通过邮件中的提示注入成功让 Agent 修改自身配置。信息窃取器（Vidar 等）已将 `~/.openclaw/` 路径列为必窃目标。

## 关键洞察

"自我修改软件"听起来像科幻，但本质上是一个**自举（bootstrapping）问题**：一个系统修改自身的能力既是最强大的特性，也是最危险的攻击面。OpenClaw 通过 250ms debounce 热重载让这种修改"即改即生效"，但没有提供"改错了怎么回滚"的机制。这意味着一次失败的自我修改可能让 Agent 陷入**不可恢复的退化状态**——除非操作者手动介入恢复文件。
