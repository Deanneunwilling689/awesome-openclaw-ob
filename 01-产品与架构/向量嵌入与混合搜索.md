# 向量嵌入与混合搜索

## 一句话总结

OpenClaw 的记忆检索使用 SQLite + sqlite-vec + FTS5 构建了一套纯本地的混合搜索引擎——语义搜索（向量相似度）占 70% 权重捕获"意思相近"的内容，关键词搜索（BM25）占 30% 权重锁定精确术语，两路结果取并集后加权排序。

## 核心机制：三个技术组件

### sqlite-vec：向量存储与相似度计算

sqlite-vec 是 SQLite 的向量扩展，让一个轻量级嵌入式数据库也能做向量搜索：
- 文本通过嵌入模型转为固定维度的浮点数组（向量）
- 向量存储在 SQLite 数据库 `~/.openclaw/memory/<agentId>.sqlite` 中
- 查询时计算余弦相似度，返回语义最接近的结果

### FTS5：全文搜索与 BM25 评分

FTS5 是 SQLite 内置的全文搜索引擎，实现 BM25 关键词匹配：
- 对文本建立倒排索引
- BM25 算法根据词频和文档频率计算相关性分数
- 精确匹配：搜索错误码 `ERR_CONNECTION_REFUSED` 或函数名 `transformContext` 时，关键词搜索远优于语义搜索

### 两路融合：70/30 加权公式

```
finalScore = 0.7 * vectorScore + 0.3 * textScore
```

**为什么是 70/30**：Agent 的大部分查询是自然语言（"用户之前说过关于旅行的偏好"），语义匹配更重要。但 30% 的关键词权重确保了精确查找不丢失——比如搜索特定的错误日志或配置项名称。

**关键设计**：使用 **union（并集）** 而非 intersection（交集）。即使某个结果只在向量搜索中得分高、在关键词搜索中得分为零，它仍然被纳入最终排名。这保证了召回率。

## 后处理优化

- **MMR（最大边际相关性）**：去重，避免返回一堆语义重复的结果
- **时间衰减**：可选，半衰期默认 30 天，让近期记忆更容易被找到
- **优雅降级**：嵌入不可用时自动退回纯 BM25，绝不硬崩溃

## 在 Agent 执行循环中的位置与本地优先设计

混合搜索在[[Agent-Flow-Loop 原理]]的 Phase 3（Context Assembly）中触发。搜索结果作为[[记忆系统]]的输出，与系统提示、Skill 定义等一起组装成完整的上下文。

OpenClaw 选择 SQLite 而非 Pinecone/Weaviate 体现了**本地优先（local-first）**理念：零外部依赖、单文件部署、离线可用、数据从不离开用户设备。

## 关键洞察

70/30 的权重比并非随意选择，它反映了 Agent 使用场景的本质：大部分时候你在问"我之前说过什么类似的话"（语义），偶尔你在找"那个具体的错误码是什么"（关键词）。纯向量搜索会丢失精确术语，纯关键词搜索看不到同义词——只有混合搜索能两全。这也是为什么 OpenClaw 在记忆检索上的表现优于只用关键词匹配的传统方案。
